# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`mathbox-react` is a React wrapper for [MathBox](https://github.com/unconed/mathbox) (a WebGL/Three.js math visualization library). It lets you build MathBox scenes declaratively as a React tree instead of imperatively via MathBox's chained selection API.

## Repository layout

Yarn 3 monorepo (workspaces under `packages/*`), orchestrated with [Turborepo](https://turbo.build/):

- `packages/mathbox-react` — the published npm package (the real code)
- `packages/examples` — a Vite app demonstrating usage (private, not published)
- `packages/eslint-config` — shared ESLint config (`@mathbox-react/eslint-config`)
- `packages/tsconfig` — shared TS config (`@mathbox-react/tsconfig`, `base.json`)

## Commands

Use **Yarn, not npm** (`packageManager: yarn@3.3.1`). Node is pinned to `24` (`.nvmrc`).

Root scripts run the matching task across all workspaces via Turbo:
- `yarn lint` / `yarn test` / `yarn build`
- `yarn examples` — builds the package, then runs `build-watch` + the examples Vite dev server concurrently (via `scripts/run_examples.sh`). This is the main way to see rendered output during development.

Inside `packages/mathbox-react`:
- `yarn test` — Vitest (`vitest run`). Single file: `yarn test src/components/components.spec.tsx`. Single test: add `-t "test name"`. `yarn test:watch` for watch mode.
- `yarn lint` — `eslint "**/*.ts?(x)"`
- `yarn build` — emits both ESM (`build/esm`) and CJS (`build/cjs`) via two tsc passes (`tsconfig.esm.json` / `tsconfig.cjs.json`); stories and test files are excluded from ESM output.
- `yarn storybook` — Storybook on port 6006 (stories live in `src/stories`).

CI (`.github/workflows/tests.yaml`) runs `yarn lint` and `yarn test` on PRs to `main`. `.pre-commit-config.yaml` adds prettier, ruff, shellcheck, detect-secrets, etc.

## Architecture

### Component factory
Almost every exported component (`Cartesian`, `Grid`, `Point`, `Surface`, `View`, …) is generated by `mathboxComponentFactory(type)` in `src/components/components.tsx`. There is **one factory**, not hand-written components — to change how all MathBox primitives behave as React components, edit the factory. Props are strongly typed from MathBox's own `Props[T]` types.

### How rendering works
MathBox is imperative: you call `parent[type](props)` on a selection to append a node. The React layer bridges this via context:

- `MathboxAPIContext` holds the nearest enclosing MathBox **selection** (the parent node). `useMathbox()` reads it.
- `Mathbox` (root) creates the MathBox instance bound to a DOM `container`, then provides both `MathboxAPIContext` (the root selection) and `ThreestrapContext` (`selection.three`) to children. `ContainedMathbox` is a convenience wrapper that renders its own `<div>` container and mounts `Mathbox` into it.
- Each factory component, on mount (`useEffect`), reads its parent selection from context, appends its node (`parent[type](others, liveProps)`), and provides its **own** selection down to its children. On unmount it calls `selection.remove()`.

### Gotchas baked into the factory (don't "simplify" away)
- **Node creation happens in `useEffect`, and effects fire child-before-parent.** Because of this the parent in context can transiently mismatch the selection's actual parent. `isSelectionParent` (in `util.ts`) detects this and forces the node to be re-created under the correct parent. `isRootDestroyed` guards against acting on a torn-down root. See the long comments in `util.ts` for the reasoning.
- `canNodeHaveChildren` / `ParentNodeTypes` (`util.ts`) is the hard-coded list of MathBox node types that accept children; the factory uses it to type and gate the `children` prop.
- **`liveProps`** are per-frame functions `(t, dt) => value` passed as a second arg to MathBox. When they change they must be fully unbound and rebound (a prop can't be re-assigned live without first being un-assigned) — the factory handles this.
- `useDeepCompareMemo` (`util.ts`) keeps MathBox option objects referentially stable across renders so the root isn't needlessly recreated. It's intentionally a narrow use of a generally-discouraged pattern; see its comment.

### Threestrap layer (`src/threestrap`)
Escape hatch to the underlying Three.js setup that MathBox bootstraps ("threestrap"). Exported under the `mathbox-react/threestrap` subpath. `Renderer` and `Controls` are effect-only components (`return null`) that read `useThreestrap()` and mutate the renderer / OrbitControls imperatively. Only `orbit` controls are currently supported.

threestrap defaults its renderer to `THREE.WebGL1Renderer`, which three.js removed in r163. That caps the usable `three` version: the peer range is `>=0.118.0 <0.163`, and the dev dependency is pinned at `0.162.0` (the last release where `WebGL1Renderer` still exists). Bumping past that needs threestrap's renderer `klass` overridden to a WebGL2 `WebGLRenderer` and real render testing.

## Testing notes

- Vitest with the `jsdom` environment (`vitest.config.ts`), using explicit imports from `vitest` (globals are off). Vite transforms the ESM deps (`mathbox`, `threestrap`, `three`, `shadergraph`) natively, so no Jest-style `transformIgnorePatterns` is needed. `include` is scoped to `src/**` so compiled specs under `build/` are never collected.
- WebGL isn't available in jsdom. `src/testSetup.ts` stubs `HTMLCanvasElement.prototype.getContext` with a **proxy-based automock** (`testUtils.ts`) that lazily fabricates any property/method accessed. The automock is framework-agnostic (plain functions, no `vi`/`jest`). When a test hits a missing WebGL/context method, extend the automock rather than mocking MathBox itself.
- Tests assert on the real MathBox selection tree (e.g. `assertSelectionsEqual`, checking `ref.current` node identity/order), not on rendered DOM.
